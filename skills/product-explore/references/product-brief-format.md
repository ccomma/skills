# Product Brief Format

Load this when `product-explore` has already recovered the real product problem and is ready to converge into its durable handoff artifact.

## When To Use

Use this format only after the exploration has already clarified the core problem, target users, main trade-offs, and recommended direction.

Do not use it as an intake questionnaire. It is a convergence artifact, not a discovery substitute or a shortcut around the earlier reframe.

## Required Structure

```text
Status:
Evidence level:
Problem statement:
Target users:
Core value proposition:
Key design decisions:
Boundaries:
Success signals:
Constraints and assumptions:
Validation questions:
Recommended next workflow:
```

## Required Quality Bar

- `Problem statement` names the user problem, not just the proposed feature.
- `Target users` are concrete enough to guide downstream planning.
- `Key design decisions` reflect actual choices made during convergence.
- `Boundaries` say what is explicitly not included yet.
- `Success signals` use facts when available and assumptions when evidence is missing.
- `Validation questions` stay open instead of pretending uncertainty is resolved.

## Optional Additions

Add only when useful:

- parked ideas
- rejected alternatives worth remembering
- placeholder metrics clearly labeled as assumptions

## Anti-Patterns

Avoid:

- turning the brief into a PRD
- listing generic aspirations instead of real decisions
- silently promoting assumptions into facts
- using the brief as execution state or current-phase authority
