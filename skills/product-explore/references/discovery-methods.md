# Discovery Method Adapters

Load this when Step 1 has already reframed the idea in product-problem terms and the default Step 2 lenses still are not enough to close the main product risk.

## When To Use

Use this collection only when the remaining risk still needs structure after the default reframe and the default Step 2 lenses.

Do not load this for ordinary fuzzy ideas, generic brainstorming, or as the default Step 2 path.

Pick the smallest adapter that fits the remaining risk. These adapters deepen risk discovery; they do not replace the main workflow's job of recovering the real product problem before planning. Do not turn every conversation into a full product management workshop.

## Jobs-To-Be-Done Adapter

Use when the remaining risk is unclear user motivation, mixed user segments, or a product category that still hides the real progress the user wants.

Minimal output:

```text
Job statement:
Current workaround:
Triggering situation:
Desired progress:
Adoption blockers:
Evidence level:
```

Pass condition: the product is framed as user progress, not as a feature list.

## Opportunity Solution Tree Adapter

Use when the remaining risk is a crowded solution space: the user has a business outcome, but too many possible features or branches are still competing.

Minimal structure:

```text
Outcome:
Opportunities:
- [user need or pain]
Solution options:
- [option linked to one opportunity]
Assumptions:
- [risky belief to validate]
```

Keep opportunities separate from solutions. If a branch contains only solution ideas, reframe upward into the user need.

## Riskiest Assumptions Adapter

Use when the remaining risk is commitment uncertainty: the bet may fail because of desirability, usability, feasibility, viability, trust, compliance, or distribution.

Ask:

- What must be true for this to work?
- Which assumption has the worst downside if false?
- Which assumption is least supported by current evidence?
- What is the smallest test that could change our mind?

Minimal output:

```text
Top assumptions:
1. [assumption] - Risk type: [desirability/usability/feasibility/viability/trust/distribution] - Evidence: [fact/assumption/unknown]
Validation test:
Decision threshold:
```

## Segment Slice Adapter

Use when the remaining risk is beachhead ambiguity: the product is trying to serve everyone and still lacks a credible first segment.

Slice by:

- user role
- situation or trigger
- frequency and pain level
- willingness or ability to pay
- operational complexity
- adoption channel
- risk sensitivity

Minimal output:

```text
Primary beachhead segment:
Why this segment first:
Segments explicitly not served yet:
What would justify expanding later:
```

## MVP Shape Adapter

Use when the remaining risk is scope inflation: the conversation jumps from idea to large scope before the smallest learning step is clear.

Compare:

- concierge/manual MVP
- prototype or mock workflow
- narrow automation
- full product path

Minimal output:

```text
Smallest learning goal:
Smallest user-visible value:
MVP shape:
What this MVP intentionally does not prove:
```

## Competitive And Substitute Adapter

Use when the remaining risk is market worth: the user asks whether the idea is worth building, entering a crowded space, or displacing entrenched substitutes.

Focus on alternatives, not just named competitors:

- existing products
- spreadsheets, chat, email, manual ops, or internal tools
- doing nothing
- hiring people
- changing the process instead of adding software

Minimal output:

```text
Current substitutes:
Why users tolerate them:
Opening:
Reason this product can win:
Reason it might still lose:
```

## Evidence Discipline

Use these labels consistently:

- Fact: supplied by the user or directly observed in artifacts
- Assumption: plausible but not yet validated
- Unknown: important and currently unanswered
- Validation question: a question that needs research, data, prototype testing, or real user feedback

Never promote assumptions into facts just to make the Product Brief look complete.
